# Cross-Reference Documentation Audit: When, Why, and How to Use the Triple

## Problem Frame

The alias+slug+linkref triple is now established across the repo
(introduced as a named idiom by #96, surfaced as a term by #99, given a
normative adjacent-placement rule by #103). Every cross-reference
surface in the repo recommends or describes it. What none of them does
well is answer the **why** and **how** for a reader who already has a
working cross-reference setup of their own.

A specific customer profile surfaced this in conversation:

- Anchors are generated by an automation pipeline as
  unique-by-construction IDs with contextual prefixes (e.g.,
  `sh-ug-123456`). Uniqueness is guaranteed at mint time.
- Within-assembly authoring uses inline anchor links today:
  `[text](#sh-ug-123456)`.
- Multi-document output (cross-file assembly with
  `<!-- include: -->`) is a future goal, not the current shape.
- The author is asking the docs: "what does the triple add for me, and
  how does it map to my workflow?"

The current docs cannot answer that question without making the reader
do most of the synthesis themselves. Five concrete gaps explain why:

1. **The "slug = alias value" variant is undocumented.** Every example
   across `spec/specification.md` § 17.3, `spec/whitepaper.md` § 3,
   `spec/cross-file-link-resolution.md` Worked Examples A and B,
   `references/best-practices.md` § Semantic Cross-References,
   `references/syntax-reference.md` § Custom Aliases note,
   `examples/semantic-cross-references.md`, the SKILL.md recommended
   pattern, and the GLOSSARY.md triple entry pairs a semantic slug
   (`installation`) with an opaque numeric alias (`#200020`). For a
   workflow whose alias values are already semantic and unique (e.g.,
   `#sh-ug-installation`), reusing the alias value as the slug
   eliminates a parallel naming vocabulary and a slug ↔ alias mapping
   table. No surface mentions this is permitted, let alone recommended
   for that workflow.

2. **The rationale leans on a benefit the reader may already have.**
   The GLOSSARY.md entry's lead claim is "stable identifier that
   survives heading-text renames." The whitepaper § 3 frames the
   triple's payoff as "rename a heading, break every link" being
   fixed by "decouple the link target from the heading text."
   `examples/semantic-cross-references.md` has a dedicated section
   titled *Surviving heading renames*. The `best-practices.md`
   *Why this is the recommended idiom* bullet about slug staying
   semantic when heading text changes is in the rationale list. A
   reader who already pairs a custom alias with `[text](#alias)`
   inline anchor links already has rename-survival from the alias
   alone -- their reference is `#alias`, not `#heading-text`, so
   heading rewording does not break it. To that reader, the
   rename-survival pitch reads as "you already have this." If that
   is the headline benefit, the triple looks like a step backwards.

3. **Co-location's drift-prevention benefit is conflated with
   heading-rename safety in the prose.** The `best-practices.md`
   *Why this is the recommended idiom* list (added by #103) places
   both rationales next to each other:

   - "The slug stays semantic [...] -- the alias `#200020` is what
     binds them." (the heading-rename axis -- alias decouples link
     target from heading text)
   - "The three pieces move as a unit when a section moves." (the
     section-move axis -- co-location keeps slug, anchor, heading
     together as a unit)

   These are two different drift axes that the docs treat as a
   single benefit. A reader who already has heading-rename safety
   from their alias treats both bullets as "the rename thing I
   already have" and dismisses the co-location pitch. The
   co-location benefit is the one that actually protects against
   silent slug-target desync, which the validator cannot detect --
   the most expensive failure mode the triple guards against, lost
   in the wash.

4. **Inline anchor links (`[text](#anchor)`) are not compared
   head-to-head with the triple anywhere.** The reader's
   starting-point pattern is what the new idiom replaces, but no
   surface names the comparison and walks through what changes.
   `references/best-practices.md` § *Advanced Patterns → Link
   References* contrasts general-purpose link references with the
   triple's link reference, but the comparison is between two
   *reference-style* forms; the reader coming from inline anchor
   links does not find their pattern in either column.
   `references/syntax-reference.md` § *Custom Aliases* shows the
   inline-anchor form (`[Introduction](#introduction)`) as the
   canonical alias-using pattern, then mentions the triple as a
   note, without contrasting the two forms.

5. **The agent/pipeline authoring audience is not surfaced.** An
   automation pipeline that mints unique anchors gets a specific win
   from slug = alias: one identifier per heading, no parallel
   vocabulary the agent must track, no mapping table to keep
   synchronized, no diff-time hazard where the linkref drifts from
   the alias. The repo has explicit AI-agent surfaces (the SKILL,
   the references), and the issue lineage (#96, #99, #103) shows
   the team treats AI agents as a load-bearing audience. The
   triple's natural fit for that audience is not stated anywhere.

The cost of all five gaps lands on the same reader: they have a
working setup, they read the docs, they cannot tell what the triple
adds for them, so they either skip it (losing future-proofing for
cross-file assembly) or adopt it without conviction (and the
authoring agent later drifts on the placement rule because the why
never landed). The fix is not to rewrite the spec or change behavior;
the spec text is correct. The fix is to make each surface answer the
**when / why / how** for a reader whose starting position is
already-working anchors and inline links.

## Requirements

### R1. Restructure `references/best-practices.md` § *Semantic Cross-References on Topic-Defining Headings* as the canonical treatment

This is the surface every other surface links to for "the full
rationale." It is the right home for the deep edit. The other
surfaces get lighter touches and rely on this section for depth.

Three structural changes within the existing section:

**R1.a. Split the "Why this is the recommended idiom" list into
labeled benefit groups.** The current flat list mixes
cross-context-resolution benefits, two distinct anti-drift benefits,
and an intent-signal benefit, so a reader skims them as one
indistinguishable pile. Group them under three internal labels so a
reader who already has one benefit can see which others are still on
offer:

- *Cross-context resolution* -- the same reference resolves in
  standalone preview, single-file publishing, and multi-file
  assembly. (existing bullet 1)
- *Anti-drift* -- two distinct axes, named explicitly:
  - *Heading-rename drift* -- the alias decouples link targets from
    heading text. **Note that an alias plus inline link gets this
    benefit on its own; the triple's contribution is to extend the
    benefit to assembly-wide reference-style links and to readers
    who rely on the slug as the human-readable handle.**
  - *Section-move drift* -- co-location keeps the directive, the
    heading, and the link reference definition together as a unit
    when a section moves, is deleted, or is reordered. The
    validator cannot detect a silent slug-target desync; the layout
    is what prevents it. **This benefit is the one inline anchor
    links and bottom-of-file linkref tables cannot give you.**
- *Intent signal* -- a custom alias paired with a linkref expresses
  that the heading is meant to be externally referenceable. An
  alias on its own can be internal or external; the paired linkref
  removes the ambiguity. (existing bullet 3, kept)

The labeling matters because the customer's specific failure mode is
reading bullets 2 and 4 of the current list as the same benefit. The
labeled split forces them apart.

**R1.b. Document the two slug variants as a recommendation, with a
rule of thumb.** Add a *Choosing the slug* subsection immediately
after *The pattern combines three pieces* block, before *All three
pieces appear adjacent*. Two variants, both conformant; choice
depends on whether the alias value is already semantic and unique:

- *Semantic slug + opaque alias* (hand-authored, publishing-tool
  numeric IDs):

  ```markdown
  <!-- style:Heading2; #200020 -->
  ## Installation

  [installation]: #200020 "Installation"
  ```

  The semantic slug provides the human-readable handle; the alias is
  opaque. Use this variant when alias values come from a publishing
  tool's numeric ID space, or any source where the value carries no
  semantic meaning on its own. This is the canonical hand-authored
  shape.

- *Slug = alias value* (generated anchors, automation pipelines,
  any unique-by-construction semantic ID):

  ```markdown
  <!-- style:Heading2; #sh-ug-installation -->
  ## Installation

  [sh-ug-installation]: #sh-ug-installation "Installation"
  ```

  When the alias is already a stable, unique, contextually
  meaningful identifier (e.g., a section ID with a workflow prefix
  minted by an automation pipeline), reusing the alias value as the
  slug avoids a parallel naming vocabulary and removes the slug ↔
  alias mapping the author would otherwise track in their head.
  Both the link reference definition and any `[text][slug]`
  reference use the same identifier.

A short rule of thumb closes the subsection: "If the alias is opaque
(numeric or otherwise non-semantic), pair it with a semantic slug. If
the alias is itself semantic and unique-by-construction, the slug =
alias variant is the cleaner fit -- fewer identifiers to keep in sync,
and authoring agents can treat the alias as the single source of
truth for the heading."

**R1.c. Add an *Inline anchor links vs. the triple* comparison
subsection.** Place it immediately before *When the rule fires*, so a
reader who reaches the application advice has already seen what they
are choosing against. Short, comparative table or paragraph -- not a
full new section. The comparison covers:

| Concern | Inline anchor link `[text](#alias)` | Triple `[text][slug]` + linkref |
|---------|-------------------------------------|--------------------------------|
| Within-document linking | Works | Works |
| Cross-file (assembled) linking | Works only if every reference knows the target's alias | Works -- the slug resolves at document-global scope after assembly |
| Heading-rename safety | Yes (alias decouples from heading text) | Yes (alias decouples from heading text) |
| Section-move safety | Reference at one site; alias on heading at another -- a section move can silently leave references pointing at nothing | Yes -- adjacency keeps directive, heading, linkref together |
| Default link text | Author writes it inline at every call site | Linkref's trailing title supplies a default; `[][slug]` and `[slug][]` work |
| Authoring overhead per heading | Lower (one anchor) | Higher (one linkref per linkable heading) |
| Best fit for | Within-assembly authoring with no cross-file plans | Authoring for cross-file resolution; future-proofing for assembly; agent/pipeline workflows |

Below the table, two paragraphs:

- *If you stay within an assembly and never need cross-file
  references*, inline anchor links are sufficient. The triple adds
  the linkref overhead with no immediate payoff -- the choice is a
  judgment call about whether future cross-file output is on the
  roadmap.
- *If you author for cross-file resolution today, or expect to in
  the future*, the triple is the recommended idiom because the same
  `[text][slug]` reference resolves in every context without
  authors having to know the target's alias at the point of use.

**R1.d. Add a *For generated-anchor and pipeline workflows* callout
to the rationale.** One paragraph, after the labeled benefit groups,
naming the audience explicitly:

"When an automation pipeline mints heading anchors, the slug = alias
variant lets every heading have a single identifier. The pipeline
doesn't need to track a parallel slug vocabulary; the authoring
agent doesn't need to keep a mapping table; cross-file references
use the same identifier the anchor minting tool already assigned.
The triple's link reference definition is mechanical to add (one
line per heading, content fully determined by the alias value), so
the per-heading authoring overhead in this variant is minimal."

## Key Decisions

- **`best-practices.md` is the canonical treatment; everything
  else is a pointer.** The deep edits land in one place so the
  guidance stays coherent and so a future change has one home to
  maintain. Every other surface links to that home. This mirrors
  the lineage established by #96/#99/#103: the prescriptive guide
  owns the recommendation; the spec stays informative; the SKILL
  points at the prescriptive guide.

- **Slug = alias is positioned as a *variant*, not a new pattern.**
  Both forms have always been syntactically valid -- this audit
  documents an existing degree of freedom, not a new feature. The
  framing in every surface presents the two as variants of one
  pattern, with a rule of thumb for choosing between them. This
  keeps the conceptual surface small and avoids creating two
  patterns to teach.

- **Heading-rename drift and section-move drift are distinct, named
  axes, not adjacent bullets.** The customer's failure mode is
  reading the two as the same benefit. The labeled split is the
  fix. Naming the axes also gives downstream content (the
  WebWorks brain repo, marketing materials, future skill updates)
  language to use without re-deriving the distinction every time.

- **The inline-anchor-vs-triple comparison goes in `best-practices.md`,
  not in `syntax-reference.md` or a new file.** It is a choice
  guide for readers researching whether to adopt the triple, which
  is the role of the prescriptive guide. `syntax-reference.md` is
  reference material -- the wrong place for a comparison that
  evaluates patterns. A new file would fragment the rationale.

- **The agent/pipeline audience is named in prose, not given its
  own section.** A separate section risks treating "agents" as a
  distinct user class and inviting future agent-specific
  treatment that splinters guidance. A short callout inside the
  rationale frames the audience as one of several reasons the
  variant exists, alongside hand-authored generated-anchor
  workflows.

- **The whitepaper edit is a reframe, not new content.** The
  section already covers cross-context resolution; the change is
  promoting that content to the lead and demoting rename-survival
  to a secondary benefit. No new examples, no new sections. The
  whitepaper is the most-read surface but the least-amenable to
  restructure -- a targeted reframe respects that.

- **Patch bump, not minor.** No format change, no new directive, no
  syntax change. The slug = alias variant is documentation of
  existing syntactic latitude. Per `CLAUDE.md` § Version
  Management, documentation updates map to `patch`.

## Assumptions

- **The customer's described setup is representative of a broader
  audience.** The Discord exchange surfaced one customer profile,
  but the same gap applies to any team with generated/automated
  anchors and within-assembly inline-link authoring. The audit
  treats the profile as a class, not as a one-off. If a future
  audit finds the profile rare, the slug = alias variant remains
  valid documentation of existing latitude even if the audience
  is smaller than projected.

- **No reader who has internalized the current rename-survival
  framing will be confused by the reframe.** The new framing
  retains rename-survival as a benefit; it just stops leading
  with it. A reader who came in expecting "the triple is for
  rename survival" still finds rename survival in the rationale,
  now under the *Anti-drift / heading-rename drift* label.

- **The slug = alias variant does not need its own GLOSSARY.md
  entry.** It is a variant of the triple, named contextually in
  the existing entry. Creating a separate term would fragment
  the vocabulary. If downstream content later wants to refer to
  the variant by name (e.g., "slug-equals-alias variant" or
  "uniform-identifier triple"), the naming question can be raised
  in a follow-up issue.

- **The whitepaper reframe will not destabilize the section's
  rhetorical arc.** The section ends with "One reference, every
  output context," which already aligns with the cross-context-
  resolution lead. The reframe brings the opening into agreement
  with the conclusion, rather than introducing a new claim. If
  review finds the reframe weakens the comparison-with-standard-
  Markdown contrast, the contrast can be preserved as a
  follow-up sentence rather than as the section opener.

- **No code path or processor behavior changes.** The audit is
  documentation only. The validator accepts both slug variants;
  the cross-file resolution rules are agnostic to the variant;
  the MDPP014 diagnostic continues to fire on cross-file slug
  conflicts regardless of variant. The implementation surface is
  unchanged.
